# Promptune Distribution, Beta Testing & Licensing Design

**Date:** 2026-03-14
**Status:** Approved
**Author:** kayumuzzaman

## Overview

This document specifies how Promptune will be distributed to users, how beta testing will be structured, and how the project will be licensed. Promptune is a community-driven, open-source terminal prompt enhancer for macOS.

**Guiding principles:**

- **TDD (RED -> GREEN -> REFACTOR)** — All new code follows the project's established TDD workflow. Tests first, implementation second, cleanup third.
- **SOLID compliance** — New code must maintain the project's existing A-grade SOLID compliance. No new responsibilities added to existing modules; extend via new modules/functions.
- **Non-breaking** — No changes to existing source files in `promptune/` unless absolutely necessary. Distribution is infrastructure, not application logic.
- **Existing patterns** — Follow the project's established patterns (Registry, Strategy, DI) and test conventions (pytest-mock, CliRunner, inline fixtures).

## 1. Distribution Channels

### 1.1 GitHub Install (Primary — Phase 1)

The initial distribution channel. Users install directly from the public GitHub repository.

```bash
pipx install git+https://github.com/kayumuzzaman/promptune.git
```

Or with pip:

```bash
pip install git+https://github.com/kayumuzzaman/promptune.git
```

### 1.2 PyPI (Primary — Phase 2)

Once the project stabilizes after beta feedback, publish to PyPI.

- Package name: `promptune`
- Install: `pipx install promptune` (recommended) or `pip install promptune`
- Pre-release versions (`0.x.x`) during beta, graduate to `1.0.0` when stable

### 1.3 Curl Installer Script

A one-liner installer for frictionless onboarding, hosted in the repository.

**Usage:**

```bash
curl -fsSL https://raw.githubusercontent.com/kayumuzzaman/promptune/main/install.sh | bash
```

**The script will:**

1. Check OS is macOS — exit with a clear message on other platforms (Promptune is macOS-only in Phase 1)
2. Refuse to run as root — exit with a warning
3. Check for Python 3.9+ — fail with a clear message if missing
4. Check for pipx — install it if not present (via `pip install --user pipx` + `pipx ensurepath`)
5. Install promptune from GitHub (initially), switch to PyPI when available
6. Verify installation by running `promptune --version`
7. Print next steps: how to set up shell integration (`eval "$(promptune shell-init)"`)
8. Remind the user they need an API key for at least one provider (Claude, OpenAI, or OpenRouter)

### 1.4 Homebrew

Not planned for initial release. May be added later via a custom tap (`kayumuzzaman/homebrew-promptune`) once there is sufficient traction.

## 2. Automated Releases

### 2.1 GitHub Actions Release Pipeline

A new workflow (`.github/workflows/release.yml`) triggered on git tags matching `v*`.

**Pipeline steps:**

1. Run existing CI checks (lint, typecheck, tests with coverage >= 90%)
2. Build sdist + wheel using `python -m build` (requires `build` package)
   - Python version: 3.12 (matching existing CI matrix)
3. Create GitHub Release with auto-generated changelog from commits
4. Attach built artifacts (`.whl` and `.tar.gz`) to the release
5. Publish to PyPI via trusted publisher (commented out initially)
   - To enable: register `promptune` on pypi.org, configure GitHub OIDC as trusted publisher under the project's publishing settings, then uncomment the PyPI step in the workflow

### 2.2 Release Workflow for Developers

```
1. Update version in pyproject.toml, promptune/__init__.py, and CHANGELOG.md
   (Version is defined in both pyproject.toml and __init__.py — both must match)
2. Commit: "Release vX.Y.Z"
3. Tag: git tag vX.Y.Z
4. Push: git push && git push --tags
5. GitHub Actions automatically:
   - Runs tests
   - Builds artifacts
   - Creates GitHub Release with artifacts
   - (Later) Publishes to PyPI
```

### 2.3 GitHub Release Artifacts

Each release includes:

- Source tarball (auto-generated by GitHub)
- Built wheel (`.whl`) attached as asset
- Built sdist (`.tar.gz`) attached as asset

## 4. Beta Testing Strategy

### 4.1 Phase 1 — Closed Beta (Soft Launch)

- Repository is **public from day one** (open source, MIT)
- Actively shared only with friends and colleagues — no community promotion yet
- Target: 5-10 trusted testers
- Goal: catch bugs, validate UX, smooth rough edges before wider audience

**Tester onboarding:**

1. Share repo link + install one-liner (curl or pipx from GitHub)
2. Remind testers they need their own API key for at least one provider (Claude, OpenAI, or OpenRouter)
3. Point testers to README for setup (shell-init, config)
4. Direct them to GitHub Issues (bugs) and Discussions (feedback/ideas)

**Feedback tracking:**

- GitHub Issues with `beta-feedback` label for tester-reported bugs
- GitHub Discussions for open-ended feedback and ideas

### 4.2 Phase 2 — Public Beta (Community Launch)

**Exit criteria from Phase 1 (all must be met):**

- At least 3 testers have completed full onboarding (install + shell-init + at least one enhancement)
- Zero critical/blocking bugs open
- At least 3 testers confirm successful install without maintainer assistance
- All GitHub Issues labeled `beta-blocker` are closed

**Promotion channels:**

- Reddit: r/commandline, r/zsh, r/python
- Hacker News (Show HN post)
- Dev Twitter/X

**Announcement content:**

- What Promptune does (one paragraph)
- 30-second demo GIF or video
- Install one-liner
- Link to GitHub repo

**GitHub Discussions categories:**

- **General** — introductions, questions
- **Ideas** — feature requests
- **Show & Tell** — users sharing their setups

### 4.3 Beta Feedback Loop

```
Tester installs → uses tool → files Issue (bug) or Discussion (idea)
→ maintainer triages → fixes/responds → releases new version → notifies testers
```

## 6. Hotfix & Rollback Procedure

If a broken version is released:

1. **Hotfix:** Fix the issue, bump patch version (e.g., `v0.2.1`), tag and push — the release pipeline handles the rest
2. **GitHub Release:** Mark the broken release as "pre-release" and add a note pointing to the fix
3. **PyPI (when enabled):** Use `pypi.org` to yank the broken version (marks it as not recommended but doesn't delete it)
4. **Notify testers:** Post in GitHub Discussions with the issue and fix

## 7. Engineering Standards & Constraints

This section ensures all implementation work aligns with the existing codebase's engineering standards.

### 7.1 TDD Approach (RED -> GREEN -> REFACTOR)

All testable deliverables follow the project's established TDD workflow:

| Deliverable | TDD Applicable? | Test Strategy |
|-------------|-----------------|---------------|
| `install.sh` | Yes | pytest-based tests using `subprocess.run()` — consistent with project's existing test stack |
| `release.yml` | Yes (partial) | CI job that runs `python -m build` on every PR to catch packaging regressions; full release validated by dry-run on a test tag |
| `LICENSE` | No (static file) | N/A |
| URL fixes | No (string replacement) | Automated grep check in CI to verify no "your-username" remains |
| Beta announcement | No (documentation) | N/A |
| Version sync | Yes | pytest test to verify `pyproject.toml` and `__init__.py` versions match |

**For `install.sh` — test cases (RED phase):**

Tests live in `tests/test_install.py` using pytest + `subprocess.run()` to invoke the script and assert on exit codes and stdout. This keeps all tests in the project's existing pytest ecosystem — no new test tooling required.

- `test_rejects_non_macos`: script exits with non-zero code and prints OS error message (tested by overriding `uname` in PATH)
- `test_rejects_root`: script exits with non-zero code when EUID=0 (tested by mocking EUID check)
- `test_rejects_old_python`: script exits with error when Python < 3.9
- `test_installs_pipx_if_missing`: pipx installation triggered when not found
- `test_installs_promptune`: promptune installed via pipx from GitHub
- `test_prints_next_steps`: output includes shell-init instructions and API key reminder
- `test_verifies_installation`: script runs `promptune --version` after install
- `test_idempotent_install`: re-running the script when promptune is already installed succeeds without errors (upgrades or skips gracefully)
- `test_network_failure_graceful_exit`: script prints user-friendly error when network is unavailable (e.g., pipx install fails)

**For version sync — test case:**

- `test_version_sync`: reads version from `pyproject.toml` and `promptune/__init__.py`, asserts they match. Added to `tests/test_cli.py` alongside existing version tests.

### 7.2 SOLID Principles — Impact Analysis

This work is **infrastructure-only** (scripts, CI config, static files). It does not modify any existing source code in `promptune/`. SOLID impact:

| Principle | Impact | Notes |
|-----------|--------|-------|
| **SRP** | No violation | New files have single responsibilities: `install.sh` = installation, `release.yml` = release automation, `LICENSE` = legal |
| **OCP** | No violation | No existing modules modified. New files are additive. |
| **LSP** | No impact | No class hierarchies involved |
| **ISP** | No impact | No interfaces involved |
| **DIP** | No impact | No dependency changes to application code |

**Key constraint:** If any future beta-driven feature requires application code changes (e.g., telemetry, feature flags), those changes must:
- Follow the existing Registry/Strategy/DI patterns
- Add new modules rather than modifying existing ones (OCP)
- Keep single responsibility per module (SRP)
- Write tests before implementation (TDD)
- Pass ruff, mypy, and coverage >= 90%

### 7.3 Non-Breaking Guarantees

**Files that WILL NOT be modified:**

- `promptune/*.py` — no application logic changes
- `tests/*.py` — no existing test modifications (new test files may be added)

**Files that WILL be modified (config only):**

- `.github/workflows/ci.yml` — coverage threshold updated from 80% to 90% to align with release pipeline (no structural changes)

**Files that WILL be modified (metadata only):**

- `pyproject.toml` — fix GitHub URLs (project.urls section only)
- `README.md` — fix GitHub URLs, add install methods

**New files to be created:**

- `LICENSE` — MIT license text
- `install.sh` — curl installer script
- `.github/workflows/release.yml` — release automation
- `tests/test_install.py` — pytest-based installer script tests

### 7.4 Design Patterns

No new design patterns are introduced. This work uses:

- **Shell scripting best practices** for `install.sh`: explicit `if` guards for each precondition check (OS, root, Python version) with user-friendly error messages printed before exit. `set -euo pipefail` is set AFTER precondition checks to allow graceful error reporting. Each logical step is a separate function (SRP at the function level).
- **GitHub Actions composition** for `release.yml`: reuse existing CI job as dependency, separate build and release jobs (SRP at the workflow level)

### 7.5 Quality Gates

All deliverables must pass before merge:

| Gate | Requirement |
|------|-------------|
| Existing CI | `ruff check .` passes |
| Existing CI | `mypy promptune/` passes |
| Existing CI | `pytest --cov=promptune --cov-fail-under=90` passes |
| Build validation | `python -m build` runs on every PR (catches packaging regressions early) |
| Package install | After `pyproject.toml` changes, verify `pip install -e .` and `promptune --version` succeed |
| Version sync | `test_version_sync` in CI ensures `pyproject.toml` and `__init__.py` versions match |
| New release workflow | Dry-run on test tag succeeds (verified by: GitHub Release created, artifacts attached, correct version in tag) |
| Install script | All `tests/test_install.py` tests pass; manual test on clean macOS environment |
| Regression | All 81 existing tests still pass with no modifications |
